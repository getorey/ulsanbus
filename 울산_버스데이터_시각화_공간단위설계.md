# 공간 단위 설계 — SGIS API 기준

> 작성일: 2026-06-15
> 관련 문서: `울산_버스데이터_시각화_자료정리.md`, `울산_버스데이터_시각화_기술스택.md`
> 범위: 배경 = 격자 / 수요특성 = 집계구 / 좌표화 = 지오코딩 (SGIS Open API 기준)
> ※ SGIS 개발지원센터가 통계청(sgis.kostat.go.kr) → 국가데이터처(sgis.mods.go.kr)로 이관 중. 엔드포인트는 운영 전 라이브 문서에서 최종 확인 필요. 아래 인증/좌표변환 엔드포인트는 확인됨, 나머지는 표준 사양 기준.

---

## 1. 설계 원칙 — 단위별 역할 분담

| 공간 단위 | 역할 | 선정 이유 |
|---|---|---|
| **격자 (Grid)** | 배경 인구밀도 히트맵 | 칸 크기 균일 → 밀도 표현·시계열 비교에 최적, 행정경계 변동 무관 |
| **집계구 (Output Area)** | 정류장 주변 수요 특성 분석 | 평균 행정동 1/29 크기, 사회경제적 동질성 → 골목 단위 수요 정밀 분석 |
| **지오코딩/좌표변환** | 좌표 ↔ 주소 ↔ 행정구역 연결 | 좌표 없는 데이터 좌표화, 점을 격자/집계구에 귀속 |

원칙: **배경은 격자, 수요 해석은 집계구, 연결은 지오코딩**. 세 가지 모두 SGIS 단일 플랫폼에서 처리해 좌표계·기준연도 정합성을 유지한다.

---

## 2. SGIS Open API 구성

- API 베이스: `https://sgisapi.kostat.go.kr/OpenAPI3/`
- 인증: 서비스 ID(consumer_key) + 보안 Key(consumer_secret)로 **accessToken 발급(유효 4시간)** → 갱신 필요
- 응답 포맷: JSON

| 구분 | 엔드포인트 | 용도 | 확인 |
|---|---|---|---|
| 인증 | `/auth/authentication.json` | accessToken 발급 (consumer_key/secret) | 표준 |
| 좌표변환 | `/transformation/transcoord.json` | 좌표계 간 변환 (posX/posY 반환) | **확인됨** |
| 지오코딩 | `/addr/geocode.json` | 주소 → 좌표(UTM-K) | 표준 |
| 지오코딩(WGS84) | `/addr/geocodewgs84.json` | 주소 → 위경도(WGS84) | 표준 |
| 역지오코딩 | `/addr/rgeocode.json` / `rgeocodewgs84.json` | 좌표 → 주소/행정구역 | 표준 |
| 인구 통계 | `/stats/population.json` | 총조사 인구(연령·성별 등) | 표준 |
| 경계(행정구역/집계구) | `/boundary/...` | 행정구역·집계구 경계 폴리곤 | 표준 |

> 좌표변환 응답 예 (확인): `{"result":{"posX":990391.6, "posY":1816238.1}, "errCd":0, "errMsg":"Success"}`

---

## 3. 좌표계 표준

| 항목 | 좌표계 | 비고 |
|---|---|---|
| GPS / 웹지도(BIS 정류장) | WGS84 (EPSG:4326) | 위경도 |
| SGIS 통계(격자·집계구) | UTM-K (EPSG:5179, GRS80) | 미터 |
| SGIS 내부 표기 | `TM_K`(UTM-K), `WGS84` 등 | 변환 src/dst 코드로 지정 |

- 변환 방향: 입력 GPS(WGS84) → UTM-K(통계 매칭) / 출력 격자 폴리곤 → WGS84(웹 렌더)
- 도구: 전처리 `pyproj`, 클라이언트 `proj4`, 대량 일괄변환은 SGIS `transcoord` API 또는 로컬 pyproj(트래픽 절감)

## 4. 단위별 처리 설계

### 4-1. 격자 (배경 인구밀도)

- 데이터: SGIS 인구 통계를 격자 단위로 취득(100m / 1km 선택)
- 매칭: GPS 점 → UTM-K 변환 → `floor(X/셀크기)*셀크기` 로 셀 ID 산출 → 인구값 조인
- 렌더: 격자 셀 경계를 WGS84 폴리곤(GeoJSON)으로 변환 → deck.gl `GridLayer`/`HeatmapLayer`
- 활용: 하루 시간축 시점 간 비교(격자 불변성 → 시계열 안정)

### 4-2. 집계구 (수요 특성)

- 데이터: 집계구 경계 + 집계구 단위 인구·가구 특성
- 매칭: 정류장 좌표 → 어느 집계구에 속하는지 point-in-polygon (GeoPandas)
- 활용: 정류장 반경 수요 특성(연령·1인가구·주택유형 등) → 트램 수요 추정 정밀화
- 주의: 집계구 경계는 기준연도에 따라 변할 수 있음 → 분석 기준연도 고정

### 4-3. 지오코딩 / 좌표변환 (좌표화·연결)

- 주소만 있는 데이터(충전소·시설 등): `geocodewgs84` 로 좌표 확보 → 격자/집계구 매칭에 투입
- 지도 클릭 → `rgeocode` 로 "OO동" 표시(툴팁)
- 좌표계 불일치 데이터: `transcoord` 로 일괄 정렬
- 정류장 좌표(BIS, WGS84)를 행정동/집계구/격자에 귀속시키는 연결 고리

## 5. 처리 파이프라인 (배치 전처리)

```
1) 인증: consumer_key/secret → accessToken (4h, 만료 시 갱신)
2) 기준 데이터 수집·캐싱
   - BIS 정류장 좌표(WGS84)        [1회]
   - SGIS 격자 인구통계            [기준연도 1회]
   - SGIS 집계구 경계 + 특성       [기준연도 1회]
3) 좌표 정렬
   - 정류장 WGS84 → UTM-K (pyproj)
4) 단위 귀속
   - 격자: floor 연산으로 셀 매칭
   - 집계구: point-in-polygon (GeoPandas)
5) 지오코딩 보완
   - 주소만 있는 데이터 → geocodewgs84
6) 산출물 export
   - 격자 GeoJSON(WGS84) / 집계구 GeoJSON / 정류장-단위 매핑 테이블(Parquet)
```

API 트래픽 절감: 격자·집계구·정류장은 변동이 적으므로 **1회 수집 후 정적 캐시**. 좌표변환은 가능하면 로컬 pyproj로 처리하고, SGIS `transcoord`는 검증용으로 사용.

---

## 6. 산출물

| 파일 | 내용 | 포맷 | 소비처 |
|---|---|---|---|
| `grid_population.geojson` | 격자 셀 + 인구밀도 (WGS84) | GeoJSON | deck.gl 배경 히트맵 |
| `output_area.geojson` | 집계구 경계 + 수요 특성 | GeoJSON | 수요 분석/하이라이트 |
| `stop_unit_map.parquet` | 정류장ID ↔ 격자ID ↔ 집계구ID ↔ 행정동 | Parquet | 조인 키 테이블 |
| `coord_lookup.parquet` | 정류장ID ↔ WGS84 ↔ UTM-K | Parquet | 좌표 변환 캐시 |
