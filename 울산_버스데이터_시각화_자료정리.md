# 울산 버스 인구이동 데이터 시각화 — 자료 정리

> 작성일: 2026-06-15
> 목적: 버스 노선별 인구 이동 데이터를 분석하여 트램 노선과의 연관성을 시각적으로 확인

---

## 1. 프로젝트 개요

버스 승하차·인구이동 데이터를 지도 위에 시각화하여, 향후 **트램 노선 후보와의 연관성**을 직관적으로 검토하는 것이 목표다.

핵심 시각화 컨셉은 다음 두 갈래로 논의되었다.

**(A) 인구밀도 히트맵 + 버스 이동경로 오버레이**
- 배경에 색깔의 진하기로 **인구 밀도**를 표현
- 그 위에 **버스 이동 경로**를 라인으로 시각화
- (옵션) 버스 이동에 따라 실제 이동하는 인구 수를 배경 색에서 **차감**하여, 순(net) 인구 이동 흐름을 파악

**(B) 개별 승객 점(dot) 애니메이션**
- 울산 BIS 승하차 정보를 지도 위에서 시각화
- 개별 승객을 **점**으로 표현하여 지도상에서 이동
- **노선별 다른 색상** 적용
- **시간 흐름**에 따라 점이 이동하고, **하차 시 사라짐**
- (확장) 개별 점 대신 **버스 덩어리(cluster)** 단위 시각화 가능성도 검토

---

## 2. 데이터 소스 명세

활용할 공공데이터 API는 두 종류다.

| 구분 | 데이터명 | 제공처 | 포맷 | 활용기간 |
|---|---|---|---|---|
| API 1 | 지역별 교통카드이용 합성데이터 | 국토교통부 | JSON / XML | 2026-06-15 ~ 2028-06-15 |
| API 2 | 울산광역시 BIS 정보 | 울산광역시 | XML | 2026-06-15 ~ 2028-06-15 |

- 공통 일반 인증키: `<공공데이터포털에서 발급받은 인증키 입력>` (보안상 공개 저장소에 키를 넣지 마세요)
- 인증키는 Encoding/Decoding 형태 중 구동되는 키를 사용

### 2-1. 국토교통부 — 지역별 교통카드이용 합성데이터

실제 승객 단위의 승하차 이동(OD) 데이터로, **인구 이동 흐름 분석의 핵심 소스**다.

- End Point: `https://apis.data.go.kr/1613000/RegionalTransportationCardUsageSyntheticData`
- **오퍼레이션: `/getUlsanTransportationCardUsageSyntheticData`** (울산)
- 참고문서: `OPENAPI 활용가이드_지역별 교통카드이용 합성데이터_v1.2`
- 데이터 링크: https://www.data.go.kr/data/15154079/openapi.do
- **상태: 라이브 호출 검증 완료(2026-06-15, resultCode 200 SUCCESS)** — 키 정상 작동

**요청 파라미터** (※ 호출 검증 결과 opr_ymd·rte_id·users_type_cd·ride_ctpv_cd **모두 필수**. 누락 시 `code 52 MISSING_REQUIRED_PARAMETER`)

| 항목 | 필수 | 설명 | 예시 |
|---|---|---|---|
| serviceKey | ● | 서비스 인증키 | (인증키) |
| pageNo | ● | 페이지 번호 | 1 |
| numOfRows | ● | 한 페이지 결과 수 | 1000 |
| dataType | ● | 데이터타입 | JSON / XML |
| opr_ymd | ● | 운행일자 | 20260101 |
| rte_id | ● | 노선아이디 | 31001122 |
| users_type_cd | ● | 이용자유형코드 | 01 |
| ride_ctpv_cd | ● | 승차시도코드(울산=31) | 31 |

> **검증된 호출 예시** (54건 반환):
> `.../getUlsanTransportationCardUsageSyntheticData?serviceKey=KEY&pageNo=1&numOfRows=1000&dataType=JSON&opr_ymd=20260101&rte_id=31001122&users_type_cd=01&ride_ctpv_cd=31`
>
> **응답 래퍼**: `Response.body.items.item[]` (배열), `Response.body.totalCount` 로 페이지네이션. 헤더 `Response.header.resultCode`.
>
> **중요(수집 설계)**: `rte_id`가 필수이므로 **노선 목록을 먼저 확보**(BIS 노선정보)한 뒤 노선별로 순회 수집해야 함. 이용자유형(`users_type_cd`)도 코드별로 나눠 호출.

### 2-2. 울산광역시 BIS 정보

정류장 좌표·노선 형상 등 **지도 매핑에 필요한 기준 정보(좌표)**를 제공한다.

- End Point: `https://openapi.its.ulsan.kr/UlsanAPI/`
- 참고문서: `OpenAPI활용가이드_울산광역시_BIS_v4.2`
- 포맷: XML
- 일일 트래픽: 각 기능당 1,000회

**활용신청 상세기능**

| No | 상세기능 | 주요 제공 정보 |
|---|---|---|
| 1 | 노선별 정류장정보 조회 | 노선ID, 정류장 순번, 정류장ID, 정류장명 |
| 2 | 버스정류장정보 조회 | 정류장ID, 정류장명, 노선유형, **경도·위도**, 비고 |
| 3 | 교통카드충전소정보 조회 | 상점명, 주소, 경도·위도, 유형 |
| 4 | 버스도착정보 조회 | 남은 정류장 수, 도착시간, 정류장ID/명, 현재 버스 위치, 노선번호 |

> **시각화 활용 포인트**: 기능 2(정류장 좌표)와 기능 1(노선별 정류장 순서)가 필수. 이 둘로 API 1의 정류장 ID를 지도상 위경도로 변환하고, 노선 경로(폴리라인)를 그린다.

### 2-3. 울산광역시 인구 데이터 (배경 인구밀도용)

히트맵 배경(인구 밀도)과 순 인구이동 분석에 쓰일 인구 통계 소스다.

**(1) 울산광역시_인구 현황 — 공공데이터포털 (현재 보유)**

- 데이터명: `울산광역시_인구 현황_20251112`
- 제공기관: 울산광역시 (법무통계담당관, 052-229-2293)
- 포맷: CSV / 전체 122행 / 업데이트 주기: 연간 (차기 등록 2026-11-12)
- 링크: https://www.data.go.kr/data/15005995/fileData.do

> **중요**: 이 데이터는 "인구 현황"이라는 이름과 달리 실제로는 **인구 이동(전입·전출·순이동) 통계**다. 컬럼 구성: 행정구역코드, 행정구역명, 행정구역레벨(시도/시군구/읍면동), 성별, 내국인(인구수), 총 이동 전입/전출, 시군구내 전입/전출, 시군구간 전입/전출, 시도간 전입/전출, **순이동**. → 본 프로젝트의 "인구 이동 흐름" 주제와 직접 연결되며, **순이동(net) 차감 시각화의 기준값**으로 활용 가능.

### 2-4. 공간 분석 단위 개념 (집계구 · 격자 · 지오코딩)

배경 인구밀도와 좌표 매핑의 정밀도를 결정하는 핵심 공간 단위 개념. 행정구역(시·도 > 시·군·구 > 읍·면·동)은 분석하기엔 너무 넓거나 인구 불균형이 심하므로, 아래 단위를 함께 활용한다.

**(1) 집계구 (Output Area) — 사회경제적 동질성을 지닌 최소 통계 단위**

- 정의: 통계청이 만든 최소 통계 구역. 같은 행정동 안이라도 거주 성향이 다른 블록을 구분.
- 기준: 인구 규모(최적 500명 내외), 사회경제적 동질성(주택 유형·공시지가), 지형지물(도로·하천 등).
- 크기: 평균적으로 행정동의 약 **1/29** 크기로 매우 촘촘함.
- 장점: "이 블록은 아파트 고소득 가구", "저 블록은 빌라 20대 1인 가구"처럼 **골목길 단위 세밀한 타게팅 분석** 가능.
- 본 프로젝트 활용: 정류장 주변 수요 특성(거주 인구 성향)을 집계구 단위로 분석 → 트램 수요 추정 정밀화.

**(2) 격자 경계 (Grid) — 행정구역을 초월한 표준 사각형 단위**

- 정의: 국토 전체를 바둑판처럼 정사각형(예: 100m×100m, 1km×1km)으로 균일하게 분할.
- 기준: 행정 경계·인구수와 무관하게 오직 '공간적 크기'만 봄.
- 장점 1 (히트맵): 모든 칸 크기가 동일해 **인구 밀도 히트맵** 표현에 가장 적합 → 본 프로젝트의 배경 밀도 레이어에 최적.
- 장점 2 (시계열): 신도시 개발로 행정동·집계구 경계는 바뀌어도 **격자는 불변** → 시점 간 정확한 비교(시계열 분석)에 유리.
- 본 프로젝트 활용: deck.gl `GridLayer`/`HeatmapLayer` 배경 + 하루 시간축 변화의 시점 간 비교 기준.

**(3) 지오코딩 (Geocoding) — 텍스트와 공간의 연결고리**

- 지오코딩(주소 → 좌표): 자연어 주소를 위도·경도(또는 SGIS 전용 좌표계 X, Y)로 변환.
- 역지오코딩(좌표 → 주소): 지도 위 한 점을 클릭하면 해당 행정동·번지를 텍스트로 반환.
- 본 프로젝트 활용: 좌표가 없는 데이터(주소만 있는 충전소·시설 등)를 좌표화하고, 정류장 좌표를 행정동/집계구/격자에 귀속시켜 인구 데이터와 매핑. 지도 클릭 시 위치 정보 툴팁(역지오코딩) 제공.

> **단위 선택 가이드**: 배경 밀도 히트맵 = **격자**(균일·시계열 안정), 정류장 주변 수요 특성 = **집계구**(동질성·정밀), 좌표↔주소↔행정구역 연결 = **지오코딩**. SGIS API가 세 단위를 모두 지원하므로 한 소스로 통합 가능.

### 2-5. 좌표계 · 격자 매칭 (GPS 좌표를 격자 통계 지도에 표시)

GPS 좌표(정류장·버스·승객)를 격자 인구통계 배경 위에 정확히 표시하는 방법. 핵심은 **좌표계 변환 → 격자 셀 매칭** 두 단계다.

**좌표계 차이**

| 데이터 | 좌표계 | 단위 |
|---|---|---|
| GPS / 웹지도(BIS 정류장 등) | WGS84 (EPSG:4326) | 위도·경도(度) |
| SGIS 격자·집계구 통계 | UTM-K (EPSG:5179, GRS80) | 미터(m) |

두 좌표계가 다르므로 그대로는 겹쳐지지 않는다 → 반드시 변환 필요.

**처리 흐름 (점 → 격자 셀 매칭, point-in-grid)**

```
GPS 좌표(위도·경도, EPSG:4326)
  → UTM-K(EPSG:5179)로 투영 변환  (X, Y 미터)
  → 셀 크기로 내림 연산
       cellX = floor(X / 100) * 100      # 100m 격자 예시
       cellY = floor(Y / 100) * 100
  → 격자 셀 ID 산출 → 해당 셀의 인구통계 값 조인
  → 지도에 표시
```

- polygon 교차 없이 **산술 연산만으로** 빠르게 처리(대량 점에 유리)
- 셀 크기(100m / 1km)는 분석 목적에 맞춰 선택

**지오코딩 연계**

- 주소만 있는 데이터(충전소·시설 등) → **지오코딩(주소→좌표)**으로 좌표 확보 후 위 흐름에 투입
- 지도 클릭 시 "여기는 OO동" 표시 → **역지오코딩(좌표→주소)**
- SGIS API가 좌표 변환·지오코딩·격자 통계를 모두 제공 → 단일 소스로 연결 가능

**렌더링 주의사항**

- deck.gl / MapLibre는 내부적으로 **WGS84로 렌더링**한다. 따라서 격자 셀 경계 좌표(EPSG:5179)를 **다시 WGS84로 역변환**해 폴리곤으로 그려야 GPS 마커와 정확히 정렬됨.
- 변환 라이브러리: `proj4`(JS), `pyproj`(Python 전처리). 전처리 단계에서 격자 GeoJSON을 미리 WGS84로 변환해두면 클라이언트 부담 감소.

> 결론: 변환할 좌표계만 일치시키면 정류장·버스·승객 점을 **인구밀도 격자 배경 위에 정확히 오버레이**할 수 있다(표준 작업).

## 3. 데이터 결합 구조

두 API는 **정류장 ID**를 매개로 연결된다.

```
[국토부 교통카드 데이터]              [울산 BIS]
 RIDE_STTN_ID / GOFF_STTN_ID  ──매핑──▶  정류장ID → (경도, 위도)
 RTE_ID                       ──매핑──▶  노선ID  → 노선명/색상/정류장 순서
 RIDE_DT / GOFF_DT            ─────────▶  시간축(애니메이션 타임라인)
 UTZTN_NOPE                   ─────────▶  인원수(밀도/차감 계산)

[인구 데이터]                         [지도]
 SGIS 격자 인구밀도            ─────────▶  배경 히트맵(정적 밀도)
 울산 인구현황(행정동 순이동)   ──매핑──▶  행정구역코드(행정동) → 순(net) 이동 차감
 정류장 좌표 → 행정동 polygon  ─────────▶  버스 통행량을 행정동 단위로 집계
```

처리 흐름:
1. BIS 기능 2로 정류장 ID→좌표 룩업 테이블 구축
2. BIS 기능 1·6으로 노선별 정류장 순서·노선 메타 확보
3. 국토부 데이터의 각 승하차 레코드를 좌표로 변환
4. 시간(`RIDE_DT`~`GOFF_DT`) 기준으로 점의 출발·소멸 타이밍 산출

## 4. 시각화 구현 방안

### 4-1. 기술 스택 (권장)

- **지도 엔진**: deck.gl + MapLibre GL (대량 점/라인 렌더링에 최적, WebGL 기반)
- **대안**: Mapbox GL JS, Kepler.gl(코딩 최소화 프로토타입용)
- **데이터 처리**: Python(pandas)로 전처리 → GeoJSON/Arrow로 export
- **타임라인 애니메이션**: deck.gl `TripsLayer` (시간축 기반 경로 이동에 최적)

### 4-2. 레이어 구성

**① 인구밀도 배경 (Heatmap / Grid)**
- deck.gl `HeatmapLayer` 또는 `GridLayer`/`HexagonLayer`
- 승차 인원(`UTZTN_NOPE`)을 정류장 좌표에 누적 → 색 진하기로 밀도 표현
- 색상: 단색 순차 컬러맵(예: 연한 노랑→진한 빨강)

**② 버스 이동 경로 (Path)**
- deck.gl `PathLayer` 또는 `TripsLayer`
- BIS 노선 형상 기반 폴리라인, **노선별 색상** 구분
- 경로 두께를 통행량에 비례시켜 주요 축 강조

**③ 개별 승객 점 (Animated Dots)**
- `TripsLayer` 또는 `ScatterplotLayer` + 시간 보간
- 승차 시각에 점 생성 → 경로 따라 이동 → 하차 시각에 소멸
- 점 색 = 노선 색, 시간 슬라이더로 재생/정지 제어

**④ (옵션) 순 인구이동 차감 모드**
- 배경 밀도에서 "현재 버스로 이동 중인 인원"을 실시간 차감
- 정류장 잔류 인구의 순(net) 변화를 색으로 표현 → 유입/유출 흐름 가시화

### 4-3. 인터랙션

- 시간 슬라이더(시:분 단위 재생, 배속 조절)
- 노선 on/off 토글 (특정 노선만 보기)
- 모드 전환: 히트맵(A) ↔ 점 애니메이션(B) ↔ 버스 덩어리(cluster)
- 정류장 hover 시 승하차 통계 툴팁

### 4-4. 트램 연관성 분석 연결

- 통행량 상위 OD축을 추출 → 지도에 굵은 라인으로 강조
- 이 주요 축과 **트램 노선 후보**를 오버레이하여 정합성 비교
- 정량 지표: 노선 구간별 총 통행량, 평균 이용거리(`UTZTN_DSTNC`), 시간대별 피크

### 4-5. 하루 단위 시간축 — 인구 이동 흐름 재생

핵심 요구사항: **하루(24시간) 단위의 시간 변화에 따라 인구 이동 흐름을 확인**할 수 있어야 한다.

- 타임라인은 00:00 ~ 24:00을 한 사이클로 하며, 슬라이더/재생 버튼으로 시간 흐름 제어
- 승차 시각(`RIDE_DT`)에 점·흐름이 생성되고 하차 시각(`GOFF_DT`)에 소멸 → 시간대별 이동량 변화가 누적·재생됨
- 배속(1×/10×/60× 등) 및 특정 시각으로 점프 가능
- 출근(07~09시)·퇴근(18~20시) 피크 구간에서 흐름이 진해지도록 표현
- (옵션) 행정동 순이동 데이터와 결합해, 시간 경과에 따른 유입/유출 누적을 색으로 차감 표시

### 4-6. 주야(晝夜) 연출 — 해 뜨고 지는 시간 변화

요구사항: 하루 시간의 변화는 **해가 뜨고 지는 시간 변화에 따라** 시각적으로 확인할 수 있어야 한다.

- 시간축 진행에 맞춰 지도 배경 밝기·색온도를 **일출~일몰 사이클**로 전환
  - 새벽(어두움) → 일출(주황빛) → 낮(밝음) → 일몰(붉은빛) → 밤(어두움)
- 울산 기준 실제 **일출·일몰 시각**을 반영(계절별 변동 가능, 기본값은 분석 대상일 기준)
  - 구현 시 일출/일몰 시각을 파라미터화(예: 천문 API 또는 고정 테이블)
- 야간에는 점/경로의 발광(glow) 효과를 강조해 이동 흐름 가시성 확보
- deck.gl 환경에서는 `LightingEffect`/`AmbientLight`의 광량을 시간에 따라 보간하거나, 배경 오버레이의 투명도·색을 시간 함수로 제어
- UI: 현재 시각과 함께 태양 위치 아이콘(해/달)으로 주야 상태를 직관적으로 표시

### 4-7. 디자인 컨셉 적용 (테마)

요구사항: 별도의 **디자인 컨셉을 적용할 수 있도록** 구조를 분리한다.

- 색상·폰트·노선 색 팔레트·배경 스타일을 **테마 토큰(JSON/CSS 변수)**으로 분리해 교체 가능하게 설계
- 지도 베이스맵 스타일(라이트/다크/커스텀) 교체 지원 (MapLibre style spec)
- 레이어 시각 속성(점 크기, 라인 두께, 컬러맵, glow 강도)을 설정값으로 외부화
- 주야 연출 색상 역시 테마에 포함해 컨셉별로 다르게 적용 가능
- 산출물은 컨셉 시안(무드/색감)에 맞춰 일관되게 스타일링

## 6. 참고 / 유의사항

- 국토부 데이터는 **합성데이터**(실제 개인정보 아님) — 분석/시각화에 적합
- BIS 일일 트래픽 1,000회 제한 → 정류장·노선 정보는 1회 수집 후 캐싱 권장
- 두 API의 노선 ID 체계(`RTE_ID` vs BIS 노선ID)가 동일한지 **사전 검증 필요**
- 국토부 데이터에는 좌표가 없으므로 BIS 좌표 매핑이 시각화의 전제 조건
